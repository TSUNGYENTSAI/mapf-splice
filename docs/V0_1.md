# v0.1 Vision and Scope

## Product statement

MAPF Splice is a reference implementation of a hybrid robot-fleet execution
architecture:

> Keep independent routing and traffic admission for normal fleet operation.
> When a stable local deadlock occurs, solve only the affected group with MAPF,
> then execute that synchronized solution asynchronously through an Action
> Dependency Graph.

The project is intended for engineers who understand MAPF solvers but are unsure
how to introduce them into an existing fleet system with continuous tasks,
rolling reservations, variable action durations, and non-synchronized robots.

## v0.1 outcome

The release must produce two primary artifacts:

1. A polished, short warehouse animation that explains the problem and the
   recovery flow without requiring the viewer to read the code first.
2. A technical article that explains the system boundaries, execution model,
   safety invariants, design trade-offs, and limitations.

The code supports these artifacts, but must also stand on its own as a small,
readable reference that another engineer can modify or embed experimentally.

## Core capabilities

### Continuous fleet operation

- A small grid warehouse with shelves, aisles, pickup points, and drop-off
  points.
- A compact synthetic hero topology with three boundary handoff stations, five
  delivery berths, and three normal task routes that form a genuine local SCC.
- Calibrated bootstrap releases (`T2=0`, `T1=5`, `T3=12`) and one lower-loop
  shelf ensure the first prospective SCC that reaches stability contains all
  three plan versions for every supported K. Earlier pair observations may
  occur but do not reach the threshold before the graph expands.
- A deterministic lifelong stream of pickup-and-delivery tasks: three bootstrap
  tasks establish the hero interaction, then a seeded generator continues
  releasing work within a bounded evaluation horizon.
- A minimal dispatcher that assigns available robots.
- A complete, observable task lifecycle: pending, assigned, to-pickup,
  carrying, to-drop-off, and completed. A robot that is already at the pickup
  still passes through the applicable transitions; deterministic tick phases
  may perform more than one transition in a tick, and every transition is
  recorded in the trace.
- Single-agent A* routing for each active task phase.
- Deterministic A* tie-breaking with neighbor priority south, west, east, then
  north. The hero bootstrap routes are executable expectations rather than
  render-only route choices.

This models lifelong fleet operation, not a lifelong MAPF solver. MAPF is an
on-demand recovery mechanism within the continuous operation.

### Traffic admission

- Vertex and edge resources for point robots on a grid.
- A committed rolling action window of length `K` (the "droplet") that grants
  exclusive motion authority over upcoming resources.
- Atomic acquisition of the initial committed horizon; a partial prefix does
  not authorize cruise execution.
- An additional read-only preview horizon of `K` actions, for a total lookahead
  of `2K`, that exposes approaching conflicts without claiming resources.
- A configurable `K` range of 3–5 cells in the hero scenario; changing `K`
  changes when the same prospective SCC becomes visible, not the task routes.
- A generic, documented nominal-speed/stopping-distance model for selecting
  the committed horizon. Its values are synthetic rather than hardware fitted.
- Incremental release and replenishment of committed reservations at
  deterministic tick boundaries.
- Deterministic action-duration and wait injection to model unequal execution
  progress without a separate communication-latency model.
- Prospective dependencies when preview actions encounter occupied or committed
  resources. Preview-versus-preview overlap is diagnostic contention, not
  resource ownership or a wait-for edge.

### Deadlock recovery

- A complete prospective dependency graph and confirmed wait-for graph that can
  represent multiple blockers per robot.
- Detection of strongly connected components in both graph stages.
- A progress-aware stability check that distinguishes temporary preview
  contention from a stable cyclic risk.
- Plan-version-scoped SCC candidates become stable after a configurable number
  of consecutive preview observations; v0.1 defaults to two.
- Scoped selection of the affected group for containment and recovery.
- Containment of a stable prospective cycle by stopping further droplet
  extension and letting already authorized progress reach a deterministic
  quiescent state.
- Confirmation that the cycle still prevents required motion authority after
  quiescence before classifying it as a hard reservation deadlock.
- Scoped MAPF solving from current simulated positions to current task-phase
  goals.
- Versioned replacement of the affected robots' remaining plans.
- Unrelated active robots remain on their existing plans and continue normal
  admission and execution through the same global occupancy and reservation
  authority used by recovery.
- Every proposal is bound to the exact confirmed incident (cycle core, affected
  scope, and confirmation tick). Installation is a revalidated all-or-nothing
  version transition: it replaces the unfinished current-phase route without
  changing task ownership, payload, phase, position, or occupancy. Completion
  evidence is recorded before ordinary task-phase advancement. Installed ADGs
  use the explicit low-speed `RECOVERY_ADG_BOUNDED_PREFIX` policy; normal
  K-cell initial admission remains atomic and unchanged.
- Fail-safe behavior when recovery planning fails.

### Asynchronous MAPF execution

- Conversion of synchronized MAPF paths into explicit actions.
- Same-agent sequencing and cross-agent dependencies.
- Action Dependency Graph validation before execution.
- Dependency-aware execution under deterministic robot delays.
- Runtime occupancy checks as a final safety invariant.

### Deterministic execution model

- One authoritative simulation state; v0.1 does not separately model physical
  truth and delayed controller observations.
- A deterministic phased tick loop rather than a wall-clock event queue or
  nondeterministic robot threads.
- Movement remains one grid edge per action. Unequal speeds and disruptions are
  represented by deterministic multi-tick action durations and explicit waits,
  not by skipping cells.
- Planned waits and execution delays remain distinct: a repeated MAPF cell is a
  finite wait action, while an injected delay extends the active action's
  progress without creating a blocked or synthetic wait action.
- Admission and action starts are decided from a stable tick snapshot so robot
  iteration order cannot mutate the conflict result implicitly.
- The canonical hero forms its prospective SCC with normal-operation delay
  probability set to zero. Seeded uncertainty remains available for later
  stress scenarios and the explicit delayed ADG recovery demonstration.

### Runtime observability

- The Python kernel may attach a read-only recorder that emits a deterministic,
  schema-versioned replay made of full checkpoint snapshots.
- Runtime inspection uses generated replay frames as its only evidence source.
- The offline browser inspector is a pure consumer. It can select and render
  frames but cannot route, reserve resources, derive dependencies, calculate
  SCCs, or mutate simulation state.
- v0.1 observability covers committed and preview authority, prospective
  dependencies and contentions, SCC stability, containment, quiescence, the
  single-incident confirmed wait-for graph and its outcome, and frame-local trace
  events. Scoped MAPF recovery states appear only after that capability exists.

## Comparison modes

The planned comparison uses the same map, task stream, robot timing, and delay
schedule in both systems:

1. **Rule-based traffic**: independent A*, rolling reservations, and a small set
   of representative yielding or backtracking heuristics.
2. **MAPF Splice**: the same normal operation, plus prospective-cycle
   containment, confirmed deadlock detection, scoped MAPF recovery, plan
   replacement, with scoped recovery ADG motion authority implemented through
   `RECOVERY_ADG_BOUNDED_PREFIX`.

An idealized global-MAPF run may be included as a reference benchmark, but it is
not the main product story.

The first executable vertical slice focuses on MAPF Splice alone. The
rule-based comparison is added only after normal traffic, prospective-cycle
containment, recovery installation, and an approved ADG admission profile work
end to end; it is not a blocker
for starting the core implementation.

## Explicit non-goals

The following are outside v0.1:

- Production deployment, certification, or claims of physical safety.
- Robot footprint geometry, swept-volume computation, or heterogeneous
  kinematics.
- Idle-blocker recruitment and generated evacuation tasks.
- Distributed services, databases, networking, ROS, or VDA 5050 integration.
- Advanced dispatch optimization, charging, battery models, task dependencies,
  or storage-allocation optimization.
- Dynamic map changes and arbitrary external obstacles.
- A separate physical-state, observed-state, telemetry, or communication-delay
  model.
- A general-purpose lifelong MAPF implementation.
- Guaranteed recovery from every solvable warehouse configuration.

## Current recovery-transaction milestone

Completed here: incident-bound proposal generation, immutable replacement
specifications, stale-state validation, atomic multi-robot `v -> v+1`
installation, typed installation failures, replayable lifecycle evidence, and
explicit characterization of ordinary K-cell admission rejection.

The executable milestone implements asynchronous and delay-tolerant recovery
through an action-boundary profile distinct from cruise authority. Physical
braking enforcement remains an AMR/AGV control-layer integration assumption;
dependency-aware conditional future ownership (Option A) remains unsupported.

## Quality bar

v0.1 is complete only when:

- One documented command reproduces the hero demonstration.
- Comparison modes consume identical deterministic scenario inputs.
- No coordinated run violates the declared vertex, edge, reservation, version,
  or dependency invariants.
- Recovery visibly selects only the relevant robot subset.
- The hero characterizes why synchronized MAPF/ADG precedence does not directly
  fit independently atomic K-cell future reservation windows.
- Metrics and a machine-readable event trace are generated with the animation.
- Seeded scenario tests cover atomic recovery installation, K-admission
  rejection, and fail-safe planner failure.
- The public API and module boundaries can be understood without reading the
  visualization code.

## Open decisions before implementation

These decisions must be resolved with executable examples before the core is
implemented:

- ~~Whether recovery includes only the cycle core or an expanded affected set.~~
  Resolved: the cyclic SCC is the trigger core; containment and recovery use its
  upstream blocked closure (the affected scope). Idle-blocker recruitment and
  external-scope expansion remain out of scope.
- Which MAPF solver is the first adapter and how solver failure is represented.
  Resolved: a vendored PyPIBT adapter with typed failures (see architecture).
- ~~How non-participating moving robots are isolated from a recovery plan.~~
  Resolved: they are omitted from local MAPF and plan replacement but are not
  isolated from safety arbitration; normal and recovery traffic share the
  authoritative occupancy and committed ledger. External blocking causes a
  retryable wait, not an automatic terminal stall.
- The concrete representation and simulator API for the atomic plan-splice
  transaction. Its all-or-nothing semantics, along with the foundational ADG
  resource and dependency rules, are fixed in the architecture document.

The fair, non-strawman rule-based baseline is intentionally deferred until the
core MAPF Splice vertical slice is complete.

## Fixed-topology lifelong validation status

Completed: the checked-in three- and four-robot topologies now have a bounded,
seeded release-and-drain corpus covering repeated dispatch and task completion,
increasing plan generations, K=3/4/5, deterministic multi-tick delays, two
sequential production-runtime recoveries, an active non-participant sharing
global traffic authority, exact replay reruns, and typed safe solver failure.
Random topology generation, fairness, global liveness, and arbitrary-topology
robustness remain deferred and are not implied by these results. The remaining
v0.1 milestone is Web Inspector and presentation completion.
